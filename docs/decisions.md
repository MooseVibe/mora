## 2026-06-07 — Новые карты коммитятся как WebP без PNG-source

**Что:** Для новых pilot-карт (`ace-of-cups`, `two-of-swords`, `eight-of-pentacles`, `three-of-wands`, `six-of-swords`, `queen-of-pentacles`) в Git оставлены только production WebP-ассеты. Временные PNG-source удалены из рабочей пачки перед push, а `imageFallback` для этих карт убран из `public/assets/cards.js`. `docs/card-style.md` и `directives/04-add-tarot-cards.md` обновлены: PNG можно использовать локально для генерации и оптимизации, но не коммитить без отдельного решения.

**Почему:** UI использует WebP, а PNG-source по 2-3 МБ на карту быстро раздувают репозиторий, замедляют push/deploy и станут проблемой при расширении колоды на десятки карт. Если source-архив понадобится, его лучше хранить отдельно от production repo.

**Кто:** автор + агент.

## 2026-06-07 — Dashboard card reader открывает вытянутую карту как shared element

**Что:** Для уже вытянутой карты дня на dashboard добавлен отдельный React `DashboardCardReader`: кнопка «Развернуть» открывает full-result состояние без новой записи в `/api/draws` и без перезагрузки страницы. Открытая карта летит из блока «Ваша карта дня» в result-позицию справа, dashboard UI уходит в `opacity: 0`, текст и кнопка «Назад» появляются во время полёта. При возврате карта летит обратно в исходный слот, исходная dashboard-карта временно скрывается, чтобы не было визуального дубля, а hover/tilt применяется ко всей карте целиком.

**Почему:** Это сохраняет ритуальный shared-element опыт и не смешивает два разных сценария: новое вытягивание карты и просмотр уже сохранённой карты дня. Отдельный reader-компонент нужен как основа для будущего раскрытия recent/journal cards: он принимает source rect и данные карты, но не выбирает новую карту и не пишет в БД.

**Кто:** автор + агент.

## 2026-06-07 — Result flow скрывает dashboard UI и нормализует размер карты

**Что:** Для вытягивания карты с dashboard закреплён подход без собственного background у `#resultOverlay`: во время `body.is-drawing-card` весь dashboard UI (`.db-wrap`) уходит в `opacity: 0`, а на экране остаётся только базовый фон `body`. Result-слой, `#revealCard` и перенесённая в `body` `.deck3d` остаются поверх этого фона. Для dashboard draw-flow нормализованы целевая ширина карты в центре и финальная ширина справа, а handoff между `deck3d` и `revealCard` теперь использует фактический текущий scale, чтобы убрать скачок между сценами.

**Почему:** Собственный фон result overlay перекрывал раннюю flight-анимацию карты, а полупрозрачный/размытый dashboard оставлял видимыми header, panels, recent cards и текст под result screen. Правильная модель для этого flow: dashboard служит только стартовой точкой, затем его UI полностью исчезает, не ломая цельную flight-анимацию. Масштаб нужно считать от целевого размера и фактического scale, потому что dashboard-колода крупнее landing-колоды, а константа `1.7` давала несовпадение конца первой сцены и начала второй.

**Кто:** автор + агент.

## 2026-06-07 — Одна карта за проход и обязательная сверка с каноном

**Что:** Workflow расширения колоды изменён: один проход теперь равен одной карте, даже если пользователь просит добавить несколько карт. Перед генерацией агент должен проверить каноническую Rider-Waite-Smith композицию, после генерации сравнить результат с оригиналом, показать изображение пользователю и получить явный визуальный аппрув до оптимизации, замены ассетов и интеграции. В `docs/card-style.md` закреплено, что Mora меняет стиль, палитру и атмосферу, но не смысловую композицию карты.

**Почему:** Генерация трёх карт за один проход заняла слишком много времени и привела к пропуску визуальных ошибок, например у `six-of-swords`: лодочник выглядел стоящим в воде, а обязательная композиция лодки с шестью мечами была ослаблена. Качество колоды важнее скорости пакетной генерации, поэтому каждая карта проходит отдельную каноническую сверку.

**Кто:** автор + агент.

## 2026-06-06 — Единый источник текстов карт и локальная ротация вариантов

**Что:** React-экраны больше не держат отдельные таблицы текстов для dashboard/journal. `public/assets/cards.js` остаётся единым источником карточных текстов, изображений и `result.dayVariants`, а `src/lib/tarot.ts` стал TypeScript-адаптером для dashboard, journal, recent cards и будущего QA-preview. В draw-flow добавлена локальная история `mora:variantHistory`, чтобы при повторном выпадении той же карты на одном устройстве не показывать последний использованный вариант “Смысла карты дня”.

**Почему:** При расширении колоды нельзя поддерживать тексты в нескольких местах: это ломает consistency и усложняет будущий AI API. Ротация вариантов снижает шанс, что пользователь увидит один и тот же смысл карты два раза подряд. Полная cross-device история для авторизованных пользователей потребует отдельной миграции Supabase `card_draws.variant_idx`, поэтому текущий шаг не меняет схему БД и не рискует сломать production API.

## 2026-06-06 — Ритуальные переходы между dashboard и journal

**Что:** Для ссылок dashboard → journal и journal → dashboard добавлен точечный `RitualTransitionLink`: обычный клик показывает fullscreen-overlay с одним циклом перетасовки колоды, затем выполняет `router.push`. Overlay рендерится через React Portal прямо в `document.body`, чтобы не зависеть от layout текущего экрана. Cmd/Ctrl-click и открытие в новой вкладке остаются нативными. Сами страницы получают лёгкий fade/slide-in, а при `prefers-reduced-motion` задержка сокращается.

**Почему:** Мгновенная замена экранов технически быстрее, но ощущалась как резкий переход и ломала ритуальный характер Mora. Один короткий цикл колоды делает переход осознанным, при этом изменение ограничено двумя ключевыми ссылками и не замедляет табы дневника. Portal нужен потому, что dashboard и journal используют enter-анимации с `transform`: такой родитель может превратить `position: fixed` в слой внутри карточки или хедера, а не поверх всего viewport.

**Кто:** агент.

## 2026-06-06 — Быстрые переходы dashboard/journal и клиентские фильтры дневника

**Что:** Ссылка dashboard → journal переведена на `next/link` с prefetch. Дневник теперь один раз получает все карты пользователя на сервере, а фильтры «Все время / 30 дней / 7 дней» переключаются на клиенте без повторного запроса и без перезагрузки страницы. Для `/dashboard` и `/journal` добавлен route-level loader с маленькой колодой как fallback, если prefetch не успел.

**Почему:** Нажатие «смотреть все» и фильтры дневника не должны ощущаться как зависание интерфейса. Prefetch ускоряет переход между экранами, клиентская фильтрация делает табы мгновенными, а loader закрывает сетевую паузу в плохих условиях без изменения продуктовой механики.

**Кто:** агент.

## 2026-06-06 — Авторизованные пользователи пропускают гостевой вход

**Что:** Middleware теперь проверяет публичные точки входа `/` и `/auth`: если есть Supabase auth-cookie и сессия валидна, пользователь сразу перенаправляется в `/dashboard`. Для гостей без auth-cookie Supabase не вызывается, чтобы не замедлять первый экран. `CardSyncOnMount` дополнительно чистит устаревший `mora:pendingDraw`, если пользователь попал в dashboard напрямую на следующий день.

**Почему:** Пользователь, который уже входил через Google с того же браузера, не должен снова видеть гостевой сценарий и кнопку авторизации. При этом нельзя возвращать дорогую auth-проверку для всех гостей, иначе снова появится задержка на главной.

**Кто:** агент.

## 2026-06-06 — Перевод тяжёлых изображений карт на WebP и безопасный кеш

**Что:** Для карт таро и ключевых фоновых изображений созданы WebP-версии без уменьшения разрешения. UI переключён на WebP, при этом PNG оставлены в репозитории как fallback для логики колоды. Для `/assets/cards/*` и ключевых WebP-фонов добавлены умеренные cache headers: `max-age=86400` + `stale-while-revalidate=604800`.

**Почему:** PNG-ассеты карт весили около 51 МБ, а вместе с ключевыми фонами около 57 МБ. WebP-набор весит около 6.3 МБ, что ускоряет первый показ, результат вытягивания и повторные визиты. Долгий immutable-кеш не использован, потому что имена файлов не хешированы и будущие обновления изображений не должны залипать у пользователей надолго.

**Кто:** агент.

## 2026-06-06 — Стабилизирована инициализация гостевой карты дня

**Что:** Главная страница сначала синхронно читает `mora:pendingDraw` из `localStorage` и показывает безопасное промежуточное состояние, а проверку Supabase-сессии выполняет отдельно. Pending-карта дополнительно зеркалируется в cookie `mora_pending_draw`, чтобы серверный dashboard мог сразу показать её после логина до клиентской синхронизации с Supabase. Middleware больше не запускается на публичной главной странице, а фоновая предзагрузка карт ограничена отключённой галереей.

**Почему:** На проде гостевой экран мог сначала показывать возможность вытянуть карту, а через несколько секунд переключаться в состояние уже вытянутой карты. После Google login dashboard тоже мог сначала отрендерить пустой виджет, потому что сервер не видит `localStorage`, а клиентский sync срабатывает позже. Cookie нужна только как короткое оптимистичное состояние; Supabase остаётся источником истины.

**Кто:** агент.

**Что:** В `design-system.md` развёрнут раздел про анимации (иерархия, тайминги, easing, сценарий вытягивания карты). В `CLAUDE.md` и `project.md` добавлен принцип «анимации — часть продукта, не оформление». В чек-лист директивы редизайна добавлена обязательная проверка анимаций.

**Почему:** Главная дифференциация Mora от конкурентов — ощущение ритуала. Без проработанных анимаций (особенно вытягивания карты на 2-4 секунды и переворота) продукт превращается в обычный веб-сервис с микро-загрузками. Это противоречит сути «онлайн-таролог = заменяет визит к живому».

**Кто:** автор + агент.
