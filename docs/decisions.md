## 2026-06-11 — Share-кнопка активируется сразу после выбора share-ready карты

**Что:** В native draw-flow share-кнопка теперь синхронизируется с выбранной картой сразу после `pickCard()` / `renderResultContent()`, а не только после финального перехода в `STATE === result`. Общий click-handler по-прежнему требует `STATE === result`, поэтому во время ритуальной анимации кнопка может визуально стать active заранее, но поделиться можно только когда result-screen открыт.

**Почему:** Если выпадает карта с утверждённым `dayVariants[n].share`, пользователь не должен видеть короткий промежуточный disabled-state на уже показанном result-screen. Share-ready статус известен сразу после выбора карты, поэтому UI должен отражать его без лишнего delay.

**Кто:** автор + агент.

## 2026-06-10 — Desktop Telegram share сохраняет рабочий url-параметр

**Что:** Для desktop Telegram share URL сохраняется непустой параметр `url`, а основной текст передаётся через `text`. Markdown-разметка для жирного/курсива не используется в MVP, потому что через Telegram share URL и системный share нельзя гарантировать `parse_mode`.

**Почему:** Telegram `t.me/share/url` без непустого `url` редиректит на `telegram.org` и ломает сценарий отправки. Для MVP важнее стабильная отправка сообщения, чем полный контроль над порядком ссылки и rich-formatting. Desktop остаётся текст + ссылка без вложения картинки; картинка через Web Share API остаётся best-effort для mobile/touch.

**Кто:** автор + агент.

## 2026-06-10 — Share handler пробует прикрепить WebP карты

**Что:** Первый share handler расширен: перед вызовом `navigator.share` он пытается загрузить текущий WebP-ассет карты, создать `File` и проверить `navigator.canShare({ files })`. Если браузер поддерживает file-share, системная модалка получает картинку карты как вложение вместе с текстом и ссылкой. Если файлы не поддерживаются или загрузка ассета не удалась, flow автоматически возвращается к текстовому share / Telegram URL / copy fallback.

**Почему:** Telegram-first MVP должен по возможности отправлять саму картинку карты, но поддержка file-share зависит от браузера, ОС и Telegram-клиента. Поэтому вложение изображения сделано как best-effort слой поверх уже работающего текстового fallback, без риска сломать базовый шеринг.

**Кто:** агент.

## 2026-06-10 — Первый share handler работает без вложения файла

**Что:** Для native result-flow добавлен первый обработчик кнопки «Поделиться»: если у текущего `dayVariants[n]` есть утверждённый `share`, кнопка собирает Telegram-friendly текст, вызывает системный `navigator.share({ title, text, url })`, а при отсутствии Web Share API открывает Telegram share URL. ## 2026-06-09 — Для новых карт нужен мгновенно читаемый силуэт

## 2026-06-09 — Card reader не держит открытую карту как scale миниатюры

**Что:** `DashboardCardReader` использует FLIP-модель: overlay-карта сразу рендерится в реальном target-размере, стартует transform-ом из координат исходной плитки, открывается в `transform: none` и закрывается обратным transform к исходной плитке. На финальном handoff reader кладёт под летящую карту клон настоящей source-карты и растворяет overlay через opacity, чтобы подмена совпадала с оригинальной плиткой. Также reader заранее декодирует изображение карты через `HTMLImageElement.decode()`.

**Почему:** На больших Retina-экранах открытая карта могла выглядеть хуже, чем исходный WebP: DOM-элемент оставался размером маленькой плитки recent/dashboard и визуально увеличивался через `scale`. Для Mora качество карточного арта критично, поэтому открытое состояние должно рендерить изображение в настоящем размере, сохраняя плавность текущей shared-element анимации.

## 2026-06-06 — Ритуальные переходы между dashboard и journal

**Что:** Для ссылок dashboard → journal и journal → dashboard добавлен точечный `RitualTransitionLink`: обычный клик показывает fullscreen-overlay с одним циклом перетасовки колоды, затем выполняет `router.push`. Portal нужен потому, что dashboard и journal используют enter-анимации с `transform`: такой родитель может превратить `position: fixed` в слой внутри карточки или хедера, а не поверх всего viewport.

## 2026-06-06 — Авторизованные пользователи пропускают гостевой вход

**Что:** Middleware теперь проверяет публичные точки входа `/` и `/auth`: если есть Supabase auth-cookie и сессия валидна, пользователь сразу перенаправляется в `/dashboard`. Для гостей без auth-cookie Supabase не вызывается, чтобы не замедлять первый экран. `CardSyncOnMount` дополнительно чистит устаревший `mora:pendingDraw`, если пользователь попал в dashboard напрямую на следующий день.

**Почему:** Пользователь, который уже входил через Google с того же браузера, не должен снова видеть гостевой сценарий и кнопку авторизации. При этом нельзя возвращать дорогую auth-проверку для всех гостей, иначе снова появится задержка на главной.

## 2026-06-06 — Перевод тяжёлых изображений карт на WebP и безопасный кеш

**Что:** Для карт таро и ключевых фоновых изображений созданы WebP-версии без уменьшения разрешения. UI переключён на WebP, при этом PNG оставлены в репозитории как fallback для логики колоды. Для `/assets/cards/*` и ключевых WebP-фонов добавлены умеренные cache headers: `max-age=86400` + `stale-while-revalidate=604800`.

**Почему:** PNG-ассеты карт весили около 51 МБ, а вместе с ключевыми фонами около 57 МБ. WebP-набор весит около 6.3 МБ, что ускоряет первый показ, результат вытягивания и повторные визиты. Долгий immutable-кеш не использован, потому что имена файлов не хешированы и будущие обновления изображений не должны залипать у пользователей надолго.

## 2026-06-06 — Стабилизирована инициализация гостевой карты дня

**Что:** Главная страница сначала синхронно читает `mora:pendingDraw` из `localStorage` и показывает безопасное промежуточное состояние, а проверку Supabase-сессии выполняет отдельно. Pending-карта дополнительно зеркалируется в cookie `mora_pending_draw`, чтобы серверный dashboard мог сразу показать её после логина до клиентской синхронизации с Supabase. **Почему:** На проде гостевой экран мог сначала показывать возможность вытянуть карту, а через несколько секунд переключаться в состояние уже вытянутой карты. После Google login dashboard тоже мог сначала отрендерить пустой виджет, потому что сервер не видит `localStorage`, а клиентский sync срабатывает позже. Cookie нужна только как короткое оптимистичное состояние; Supabase остаётся источником истины.
