# Mora — Release Backlog

Этот файл — место для багов, UX-долгов и наблюдений с MVP-тестов, которые нельзя потерять до публичного релиза.

Это не `docs/decisions.md`: сюда попадает то, что нужно проверить, исправить или переосмыслить. Когда задача сделана и подтверждена визуально/продуктово, её можно перенести в changelog/decision или удалить из активного списка.

## Легенда

- **Critical** — мешает нормальному использованию или ломает ключевой сценарий.
- **High** — сильно портит опыт, но есть обходной путь.
- **Medium** — заметный UX/UI-долг.
- **Low** — полировка.

Статусы:
- `Open` — записано, не взято в работу.
- `In progress` — сейчас чинится.
- `Needs QA` — нужен ручной/визуальный тест.
- `Done` — исправлено и подтверждено.

### Critical · После посещения дневника карта дня не вытягивается с главной

**Статус:** Done  
**Дата:** 2026-06-08  
**Где:** переход `/journal` -> главная / dashboard daily draw flow  
**Источник:** MVP-тесты
**Подтверждение:** локально проверено 2026-06-08 через `/dashboard?preview=empty` → `/journal?returnPreview=empty` → `/dashboard?preview=empty`: после возврата карта вытягивается.

**Сценарий воспроизведения:**
- зайти в дневник карт;
- вернуться на главную;
- попробовать вытянуть карту;
- ничего не происходит.

**Что проверить при фиксе:**
- не остаётся ли после `/journal` состояние/класс на `body`, overlay, loader или routing transition, которое блокирует draw;
- не ломается ли инициализация `draw.js` / `TaroApp` после возврата со страницы дневника;
- не перехватывает ли клики невидимый слой;
- не конфликтует ли authenticated/dashboard flow с unauthenticated landing draw flow;
- проверить сценарий через обычную навигацию, browser back и прямой переход по URL.

**Почему важно:**
Это ломает ключевой сценарий продукта — вытягивание карты. Баг нужно держать в списке до фикса, но сейчас не чинить в рамках текущей записи.

## Open

### Medium · Dashboard card reader может микролагать при раскрытии карты

**Статус:** Open  
**Дата:** 2026-06-09  
**Где:** `/dashboard`, раскрытие карты дня и recent cards через `DashboardCardReader`  
**Источник:** ручная проверка на production, субъективное ощущение микрофризов во время полёта карты

**Что видно:**
- при клике по карте анимация раскрытия иногда ощущается не идеально плавной;
- hover/tilt на миниатюрах работает гладко, проблема заметнее именно в shared-element полёте;
- эффект может быть сильнее при VPN или первом открытии карты, когда картинка ещё декодируется браузером.

**Почему может происходить:**
- во время полёта сейчас анимируются не только `transform`/`opacity`, но и `inset`, `width`, `height`, `border-radius` у внутренней картинки;
- overlay и отдельный `<img>` создаются в момент клика, что может дать decode/paint spike;
- одновременно гасится весь dashboard через `opacity`, а на нём есть `backdrop-filter`, тени и изображения;
- большая тень летящей карты может быть дорогой для compositor на некоторых устройствах/браузерах.

**Почему важно:**
Раскрытие карты — ритуальный момент Mora. Даже небольшой frame drop ощущается сильнее, чем в обычном UI, и мешает ощущению “дорогого” плавного продукта.

**Ориентиры для будущего фикса:**
- записать Performance trace в Chrome DevTools на production: раскрытие recent card и карты дня;
- проверить FPS meter / Paint flashing, отдельно с VPN и без VPN;
- убрать анимацию `width`/`height`/`inset`, оставить движение на `transform`/`opacity`;
- заранее декодировать картинку раскрываемой карты через `HTMLImageElement.decode()`;
- уменьшить/отложить тяжёлый `box-shadow` на время полёта;
- рассмотреть заранее смонтированный reader-layer, чтобы клик не создавал весь overlay с нуля.

### Critical · Dashboard не адаптирован под разные разрешения и мобильные устройства

**Статус:** Open  
**Дата:** 2026-06-07  
**Где:** `/dashboard`, responsive/devices, особенно mobile  
**Источник:** MVP-тесты и ручная проверка через DevTools responsive mode

**Что видно:**
- dashboard хорошо выглядит только на текущем Mac-разрешении пользователя;
- при изменении ширины экрана layout заметно разваливается;
- на mobile опыт критически плохой;
- местами невозможно нормально скроллить;
- блоки dashboard не имеют полноценной responsive-адаптации.

**Почему важно:**
Mora уже даётся людям на MVP-тест. До публичного релиза dashboard должен быть пригоден не только на одном Mac-разрешении, иначе ключевой авторизованный сценарий будет выглядеть сломанным для части пользователей.

**Когда брать:**
После того как основные desktop-механики dashboard будут доведены до стабильного состояния: card draw, expand/back, recent-card reader, journal interactions. Адаптив лучше делать отдельной большой задачей, но с маленькими визуальными инкрементами и проверкой на desktop/tablet/mobile.

**Ориентиры для будущей задачи:**
- сначала зафиксировать целевые breakpoint-диапазоны;
- проверить dashboard на 1440/1280 desktop, tablet и 375 mobile;
- отдельно решить mobile navigation и scroll behavior;
- не ломать текущую быструю production-анимацию;
- все изменения проверять глазами после каждого инкремента.
