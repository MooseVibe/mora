# Mora — Agent Instructions

Ты работаешь в проекте **Mora** — это онлайн-таролог. Сейчас MVP работает, идёт фаза UI-редизайна существующих экранов.

## Что нужно знать сразу

1. **Первым делом в каждом новом чате** — прочитай `PROJECT_CONTEXT.md`. Это быстрый вход: что за проект, как запускается, где что лежит.
2. **Перед любой задачей** — прочитай `docs/project.md` (что за продукт), `docs/architecture.md` (где что лежит) и `docs/design-system.md` (визуальный язык), если задача затрагивает продукт, код или визуал.
3. **Когда работаешь над конкретной задачей** — открой нужную директиву из `directives/` и иди по шагам.
4. **После каждого значимого решения** — допиши строчку в `docs/decisions.md` (зачем сделали так, а не иначе).

Без этого ты теряешь контекст между чатами. Это главная боль, которую решают эти файлы.

## Структура папок

```
mora/
├── AGENTS.md                  ← этот файл, ты его уже читаешь
├── PROJECT_CONTEXT.md         ← быстрый старт для нового чата
├── docs/
│   ├── project.md             ← продукт, аудитория, north star
│   ├── architecture.md        ← стек, структура папок, где что лежит
│   ├── design-system.md       ← цвета, типографика, тон, визуальный язык
│   ├── features.md            ← что готово, что в работе, что планируется
│   └── decisions.md           ← лог решений
└── directives/
    ├── 01-redesign-screen.md  ← редизайн существующего экрана
    ├── 02-add-feature.md      ← добавление новой фичи
    └── 03-fix-bug.md          ← починка бага без слома механик
```

## Базовые правила работы

### 1. Не ломай рабочие механики
MVP работает: авторизация Google, вытягивание карты с сохранением, дневник карт, сводка по 3 последним картам. **Это уже работает.** Когда делаешь редизайн или добавляешь фичу — сначала пойми текущую логику, потом меняй визуал. Если непонятно как устроена механика — открой код, прочитай, не угадывай.

### 2. Держи визуальный язык
Цвета, шрифты, отступы, форма кнопок, состояния — всё это описано в `docs/design-system.md`. **Не изобретай свои стили.** Если нужного токена или паттерна там нет — сначала добавь его в дизайн-систему, потом используй. Не пиши инлайн магические значения вроде `#8B7355` или `padding: 23px` без обоснования.

### 2.1. Анимации — это часть продукта, не оформление
Mora — про ритуал, а не про быстрый клик. **Каждое значимое взаимодействие должно ощущаться как событие.** Особенно: вытягивание карты, переворот, переходы между экранами. Подробности и тайминги — в `docs/design-system.md` → раздел «Анимации».

Если ты делаешь экран и не подумал об анимации — ты сделал его наполовину. Прежде чем сказать «готово», спроси себя: *«Это ощущается как таро-ритуал или как смена страницы в админке?»*

### 3. Mobile и Desktop одинаково важны
Оба формата равноценны. Не делай экран только под одну ширину. Минимум — проверить, что вёрстка не разваливается на 375px и 1440px.

### 4. Не выдумывай данные
Если не знаешь имя файла, точку API, название переменной — открой код и посмотри. Если в коде нет — спроси пользователя. Не пиши заглушки вида `// TODO: подставить настоящий путь` без явного предупреждения.

### 5. Сохраняй контекст
После каждой задачи, в которой ты принял решение влияющее на архитектуру или визуал — допиши в `docs/decisions.md` короткую запись: что решили, почему, дата. Это нужно для следующего чата, который не помнит ничего.

### 5.1. Прод должен соответствовать git
Если изменения уже задеплоены, запушены или иным образом попали в production/preview, они должны быть закоммичены. Нельзя оставлять продовую функциональность только в незакоммиченном рабочем дереве: следующий чат должен видеть её через git history, а не угадывать по локальному diff.

### 6. Не объединяй директивы
Каждая директива решает одну задачу. Не пытайся "заодно" сделать редизайн + добавить фичу + починить баг. Это путь к мусорному коду.

## Тон общения

На «ты», без формальностей. Если что-то непонятно — спрашивай напрямую, лучше одним вопросом за раз. Если упёрся в архитектурное ограничение или нашёл проблему — говори честно, не пытайся обойти костылём.

## North Star проекта

Главная фича в перспективе — **AI-таролог в чате**, доступный когда угодно без записи. Карта дня и дневник — это retention-механика, чтобы зацепить пользователя пока AI-таролог не готов. Любое решение проверяй вопросом: *«Это приближает нас к AI-тарологу или хотя бы не мешает ему?»*
