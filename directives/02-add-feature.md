# Директива 02 · Add Feature

## Когда использовать

Пользователь хочет **новую механику** — то, чего ещё нет. Не путать с редизайном (это `01`) и багфиксом (это `03`).

---

## Предусловия

1. Прочитай `docs/project.md` — фича должна вписываться в north star (AI-таролог) или удерживать пользователя до него.
2. Прочитай `docs/features.md` — убедись что фичи ещё нет и она не в списке «не делаем».
3. Прочитай `docs/architecture.md` — пойми где будешь добавлять код.

---

## Шаги

### Шаг 1 · Бриф по фиче

Спроси пользователя:
1. **Что за фича в одном предложении?**
2. **Какую задачу пользователя она решает?** Не «потому что круто», а конкретно — что юзер не может сделать сейчас.
3. **Это часть пути к AI-тарологу или retention-механика?**
4. **MVP-версия — что минимально нужно, чтобы зарелизить?**

Один блок. Жди ответов.

### Шаг 2 · Проверка соответствия

Прежде чем писать код, проверь сам:
- Фича приближает к AI-тарологу? Если нет — это удержание пользователей? Если и нет — зачем она?
- Она не дублирует существующее?
- Не ломает ли она текущие механики?

Если есть сомнения — обозначь их пользователю и спроси.

### Шаг 3 · План реализации

Опиши:
1. Какие файлы / папки создашь или тронешь
2. Какие новые компоненты понадобятся
3. Нужны ли изменения в БД / API
4. Какие новые состояния добавятся (новые экраны, модалки, пустые состояния)

Спроси: «Так?». Жди подтверждения.

### Шаг 4 · Реализация поэтапно

1. **Логика** — сначала чистая логика без UI (функции, хуки, API-вызовы)
2. **UI-каркас** — разметка нового экрана / компонента
3. **Стили** — строго по `docs/design-system.md`
4. **Состояния** — loading, empty, error, success
5. **Интеграция** — связка с существующими экранами

После каждого этапа — чек-ин «продолжаю?».

### Шаг 5 · Проверь обязательное

- [ ] Существующие механики не сломались (карта дня, авторизация, дневник)
- [ ] Новая фича работает на 375px и 1440px
- [ ] Тексты — в тоне Mora (мистический, не маркетинговый)
- [ ] Все цвета, отступы, шрифты — из дизайн-системы
- [ ] Есть все состояния (включая ошибки)
- [ ] Если нужны новые токены — добавлены в `design-system.md`

### Шаг 6 · Зафиксируй

1. Обнови `docs/features.md` — добавь фичу в раздел «Готово» или «В работе».
2. Допиши в `docs/decisions.md`: что добавили, какие альтернативы рассматривали, почему выбрали это.
3. Если повлияло на архитектуру — обнови `docs/architecture.md`.

---

## Граница

- **Если фича большая** (например, AI-таролог в чате) — не пытайся сделать за один поход. Разбей на под-фичи, начни с самой маленькой ценной части.
- **Если фича требует подключения новых сервисов** (платежи, AI API, push-уведомления) — сначала обсуди с пользователем интеграцию отдельно, потом фичу.
- **Если в процессе обнаружил, что текущая архитектура мешает фиче** — скажи прямо, не делай костыль.
