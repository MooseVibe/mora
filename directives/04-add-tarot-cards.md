# Директива 04 · Add Tarot Cards

## Когда использовать

Пользователь хочет расширить колоду Mora: добавить новые карты, архетипы, изображения или тексты карт.

Эта директива не про редизайн экранов. Её цель — безопасно добавить карты в существующую рабочую механику: landing, dashboard, journal, recent cards и ограничение «одна карта в день».

## Предусловия

Перед началом прочитай:

1. `PROJECT_CONTEXT.md`
2. `docs/project.md`
3. `docs/architecture.md`
4. `docs/design-system.md`
5. `docs/card-style.md`
6. `docs/decisions.md`

Затем открой текущие источники карт:

1. `public/assets/cards/*.webp`
2. `public/assets/cards.js`
3. `src/lib/tarot.ts`
4. `src/components/JournalClient.tsx`
5. `src/components/RecentCardsWidget.tsx`
6. `src/components/TaroApp.tsx`
7. `src/components/DrawWidget.tsx`

## Главный принцип

Не добавляй сразу большой пакет карт.

Работай строго короткими проходами с аппрувами:

1. Один проход = одна карта.
2. Если пользователь просит 3, 5 или 10 карт — всё равно добавляй их по одной, проходя полный цикл для каждой карты отдельно.
3. Не начинай следующую карту, пока пользователь не принял текущую: канон, визуал, тексты и QA.
4. Не объединяй расширение колоды с новыми механиками вроде полного экрана вытягивания. Сначала закрой карту, потом отдельным проходом делай механику.
5. Если один проход начинает занимать больше 15 минут, остановись, дай статус и спроси, продолжать ли в текущем направлении или упростить шаг.
6. Самопроверка агента не заменяет визуальный аппрув пользователя. После генерации агент сначала сравнивает карту с каноном сам, затем показывает изображение пользователю и ждёт явное подтверждение.

## Обязательный итог для каждой новой карты

Когда карта считается добавленной, должны быть выполнены все пункты:

1. Изображение сгенерировано по `docs/card-style.md`.
2. WebP лежит в `public/assets/cards/{card-id}.webp`, оптимизирован и проверен по размеру/весу.
3. Временный PNG-source удалён из Git/worktree перед коммитом, если пользователь явно не попросил хранить source в репозитории.
4. Карта добавлена в `public/assets/cards.js` со всеми обязательными полями.
5. У карты есть минимум 3 утверждённых варианта `result.dayVariants` в формате `preview/full` для блока “Что это значит сегодня”.
6. Автор вручную прочитал и принял каждый вариант текста.
7. ID карты добавлен в `TarotCardId` в `src/lib/tarot.ts`; мета и тексты не дублируются в React-экранах.
8. Карта автоматически видна на `/qa/cards` вместе с картинкой, summary и всеми вариантами текстов.
9. Landing/dashboard/journal/recent cards не сломаны.
10. `npm run lint` и `npm run build` проходят.

Если пользователь или следующий чат просит “QA preview”, “страницу с картами и текстами”, “посмотреть карты и тексты” — дай ссылку на `/qa/cards` в активном окружении. Локально это обычно `http://localhost:3000/qa/cards`, либо тот порт, на котором поднят dev-сервер.

## Шаг 1 · Выбери одну карту

Предложи пользователю одну карту для текущего прохода. Можно назвать 2-3 кандидата, но после выбора работай только с одной картой.

Для каждой карты укажи:

- `id`
- русское название
- тип: старший аркан / младший аркан / архетип Mora
- короткий смысл
- почему эта карта полезна для daily-card retention

Если пользователь уже дал список — не спорь с ним, но подсвети риски: слишком похожие архетипы, тяжёлые темы, длинные названия, сложность визуала.

## Шаг 2 · Проверь оригинальную карту

Перед генерацией найди каноническую композицию оригинальной карты Таро, по умолчанию Rider-Waite-Smith.

Зафиксируй коротко:

- какие фигуры/объекты обязательны;
- где они расположены;
- какие символы нельзя потерять;
- что можно менять только стилистически, а не смыслово.

Пример: для `six-of-swords` в лодке должны быть пассажиры и лодочник, а шесть мечей должны стоять в лодке; нельзя делать лодочника фигурой, стоящей отдельно в воде.

## Шаг 3 · Сгенерируй одну карту

Для выбранной карты:

1. Используй `docs/card-style.md`.
2. Сохрани исходный PNG временно локально для визуального аппрува и последующей WebP-оптимизации.
3. Не меняй код и не добавляй production WebP, пока карта не прошла каноническую визуальную сверку и пользователь не дал явный аппрув.
4. Если визуал выбивается из стиля Mora или ломает канон оригинальной карты — переделай изображение, а не пытайся компенсировать это текстом.

Критерии изображения:

- тёмная гравюрная база;
- тёплый золотой смысловой свет;
- читаемый главный объект;
- вертикальный формат 2:3;
- нет названия карты на изображении;
- есть аккуратный нижний номер или согласованный символ;
- карта читается в миниатюре.
- обязательные элементы оригинальной карты сохранены в правильной логике и расположении.

## Шаг 4 · Сравни с оригиналом и покажи пользователю

Перед оптимизацией и кодовой интеграцией сравни изображение с оригинальной картой.

Покажи пользователю одну карту и кратко объясни:

- какие обязательные элементы оригинала сохранены;
- что изменено только под стиль Mora;
- где есть риск;
- нужно ли перегенерировать до интеграции.

Жди явного аппрува пользователя перед оптимизацией, добавлением WebP в `public/assets/cards/` и изменением кода. Фраза агента “мне кажется, нормально” не считается аппрувом.

## Шаг 5 · Оптимизируй изображение

Для утверждённой PNG-карты создай WebP:

```bash
cwebp -q 82 public/assets/cards/{card-id}.png -o public/assets/cards/{card-id}.webp
```

Проверь:

```bash
sips -g pixelWidth -g pixelHeight public/assets/cards/{card-id}.webp
ls -lh public/assets/cards/{card-id}.webp
```

Ориентиры:

- размер: `1024x1536`;
- WebP желательно до `350 KB`;
- максимум `500 KB` без отдельного обоснования;
- PNG остаётся временным локальным source и удаляется перед commit, если нет явного решения хранить source в Git.

## Шаг 6 · Обнови `public/assets/cards.js`

Добавь карту в `TAROT_CARDS`.

Обязательные поля:

- `id`
- `name`
- `archetype`
- `character`
- `description`
- `streetName`
- `streetText`
- `visualHint`
- `image`
- `num`
- `symbol`
- `result`

Для `result` добавь:

- `label`
- `title`
- `titleMeta`
- `tags`
- `meaningLabel`: `Что это значит сегодня`
- `dayVariants`: 3-4 утверждённых варианта в формате `preview/full`
- `streetVariants`: 3-5 вариантов, если street-mode остаётся в UI

Требования к текстам:

- `preview` — 1-2 абзаца, которые дают самостоятельный смысл карты дня;
- `full` — те же абзацы `preview` в начале плюс 1-3 дополнительных абзаца ниже;
- `preview` должен быть точным началом `full`, а не отдельным пересказом;
- классический смысл Rider-Waite-Smith вплетён в daily-reading, а не вынесен отдельным блоком “Карта в таро”;
- ясный смысл для конкретного дня;
- в тексте считывается персонаж Моры: можно использовать формулировки вроде `Мора говорит`, `Мора советует`, `Мора считает`, если это звучит естественно;
- имя Моры не нужно повторять в каждом абзаце, достаточно одного ясного присутствия персонажа в reading;
- без эзотерического шума;
- без маркетингового тона;
- варианты должны отличаться сценариями дня, а не только формулировками;
- автор вручную аппрувит каждый вариант до интеграции;
- выбор варианта должен быть стабильным для конкретной даты и карты.

Важно: в `public/assets/app.js` уже есть детерминированный выбор варианта по дате и `card.id`. Не заменяй его случайностью.

## Шаг 7 · Обнови `src/lib/tarot.ts`

Добавь ID карты в `TarotCardId`. Остальная мета, тексты для dashboard/journal и пути изображений должны подтягиваться из единого источника `public/assets/cards.js` через helpers в `src/lib/tarot.ts`.

Проверь, что `getTarotCardImageSrc(cardId)` продолжает строить путь `/assets/cards/{card-id}.webp`.

## Шаг 8 · Обнови тексты для journal/recent

Не добавляй отдельные таблицы текстов в `JournalClient.tsx`, `RecentCardsWidget.tsx` или `dashboard/page.tsx`.

Сейчас journal summary берётся из `description`, а full-result “Что это значит сегодня” берётся из `result.dayVariants` через `src/lib/tarot.ts`. Если нужен другой формат, сначала меняй общий источник/adapter, а не дублируй тексты в React-экранах.

Новый layout результата карты:

- дата сверху, приглушённо и отдельно от тегов;
- теги классификации;
- название карты + `titleMeta`;
- один главный блок `Что это значит сегодня`;
- inline-ссылка `Читать дальше`, только если `full` длиннее `preview`;
- без отдельного блока `Карта в таро` на основном result-screen.

## Шаг 9 · Проверь UI-состояния

Обязательно проверь все места, где показываются карта и смысл:

- landing после вытягивания карты;
- landing для уже вытянутой гостевой карты;
- dashboard, блок «Ваша карта дня»;
- dashboard recent cards;
- journal list;
- empty/loading states, если изменения их затронули.

Проверь ширины:

- mobile `375px`;
- desktop `1440px`.

Ничего не должно:

- вылезать за контейнер;
- перекрывать кнопки;
- ломать сетку;
- делать миниатюру нечитаемой;
- ухудшать ритуальный loader или переходы.

## Шаг 9.1 · Production QA новых карт

Когда нужно посмотреть новые карты на живом сервисе после деплоя, не добавляй временные `debugCards`, `forceCard`, `cards1/cards2/cards3` или auth-bypass параметры в рабочую механику вытягивания.

Правильный подход для будущих партий карт:

1. Использовать служебный card-preview экран `/qa/cards`, закрытый от индексации и защищённый `CARD_QA_TOKEN` в production/Vercel Preview. Локально экран доступен без токена.
2. Экран должен рендерить каждую новую карту как production result state: изображение, `result.title`, `result.titleMeta`, теги и все `dayVariants`, в тех же CSS-классах и адаптивных контейнерах, где это возможно.
3. Экран не должен писать в `localStorage`, cookie, `/api/draws` или Supabase. Это просмотр, не вытягивание.
4. Для dashboard/journal дополнительно можно проверить реальные записи через тестового пользователя и тестовые строки в `card_draws`, но только после явного аппрува, потому что это уже изменение данных.
5. После проверки на Vercel Preview или prod записать в `docs/decisions.md`, какие карты просмотрены, какие тексты/изображения приняты и что нужно поправить.

Почему так: карточный QA должен быть виден в настоящем production окружении, но не должен ломать главную механику “одна карта в день”, auth, сохранение и дневник.

## Шаг 10 · Проверь сборку

Запусти:

```bash
npm run lint
npm run build
```

Если проверка не прошла — исправь перед финальным ответом.

## Шаг 11 · Зафиксируй контекст

После добавления карт обнови:

1. `docs/decisions.md` — что добавили, почему такой состав/стиль, кто принял решение.
2. `docs/features.md` — если расширение колоды стало готовой частью продукта.
3. `docs/card-style.md` — если по итогам пилотов появились уточнения.
4. `docs/architecture.md` — только если изменилась структура данных или источники текстов.

## Граница

- Не деплой без отдельного аппрува пользователя.
- Не переписывай механику вытягивания карты без явной задачи.
- Не меняй существующие ID карт.
- Не коммить временные PNG-source без явного решения пользователя.
- Не добавляй новые UI-библиотеки ради карт.
- Не объединяй эту задачу с редизайном экранов или AI-чатом.
