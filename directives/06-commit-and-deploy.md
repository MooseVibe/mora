# Директива 06 · Commit And Deploy

## Когда использовать

Открывай эту директиву перед любым коммитом, пушем, Vercel preview или production deploy.

Цель: не выпускать изменения, которые следующий чат не сможет понять по git history и документации.

## Главное правило

Перед коммитом или деплоем документация должна отвечать на два вопроса:

1. Что уже сделано и проверено.
2. Что осталось сделать дальше.

Если изменение попало в production/preview, оно должно быть закоммичено. Прод не должен жить только в локальном diff.

## Обязательный порядок перед commit

1. Прочитай `PROJECT_CONTEXT.md` и `directives/00-start-every-task.md`.
2. Проверь `git status --short`.
3. Просмотри diff и отдели свои изменения от чужих:
   - не откатывай чужие изменения;
   - не добавляй в коммит файлы, которые не относятся к задаче;
   - если в одном файле есть чужие правки, работай с ними аккуратно и явно скажи об этом.
4. Обнови документацию, если задача изменила продукт, визуал, архитектуру, процесс или roadmap:
   - `docs/features.md` — что теперь готово / в работе / следующий шаг;
   - `BACKLOG.md` — что перенести в Done, что оставить Open, что стало ближайшим шагом;
   - `docs/architecture.md` — если поменялись структура, данные, API или ключевой flow;
   - `PROJECT_CONTEXT.md` — если изменился быстрый вход для нового чата;
   - `docs/decisions.md` — если принято значимое решение и почему.
5. Если документация не нужна, коротко объясни почему: например, `только typo`, `только тест`, `только локальный refactor без изменения поведения`.
6. Запусти минимальные проверки по риску изменения:
   - docs-only: `git diff --check`;
   - кодовая правка: минимум `npm run lint`;
   - правка механики draw/auth/save/journal/API или карточная интеграция: `npm run lint` и `npm run build`;
   - UI-правка: дополнительно визуальная проверка нужных экранов на 375px и 1440px, если возможно.
7. Покажи пользователю короткий commit summary:
   - что изменено;
   - какие docs обновлены;
   - какие проверки прошли;
   - что осталось.
8. Коммить только после явного аппрува пользователя, если он не дал его заранее.

## Обязательный порядок перед deploy

1. Убедись, что рабочее дерево чистое или в нём нет незакоммиченных изменений, которые должны попасть в deploy.
2. Убедись, что документация уже обновлена в том же коммите или в отдельном коммите перед deploy.
3. Проверь, что production/preview будет соответствовать git history.
4. Запусти проверки, достаточные для риска деплоя: обычно `npm run lint` и `npm run build`.
5. Деплой только после отдельного аппрува пользователя.
6. После deploy сообщи:
   - commit hash;
   - environment / URL;
   - что проверить глазами;
   - есть ли оставшиеся риски.

## Что нельзя делать

- Нельзя коммитить продуктовую работу, оставив `BACKLOG.md` или `docs/features.md` в состоянии “это ещё нужно сделать”, если это уже сделано.
- Нельзя деплоить незакоммиченные production-изменения.
- Нельзя смешивать в один commit несвязанные задачи, если пользователь не попросил специально.
- Нельзя чинить новый баг “заодно” во время commit/deploy-прохода. Запиши его в `BACKLOG.md` и остановись.

## Ponytail-версия

Самый короткий правильный commit/deploy-проход:

1. status;
2. diff;
3. docs/backlog в актуальное состояние;
4. минимальная проверка;
5. пользовательский аппрув;
6. commit или deploy.

Если шаг не нужен, не делай его, но назови причину.
