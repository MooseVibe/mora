# Директива 05 · Update Card Texts

## Когда использовать

Пользователь хочет актуализировать тексты уже существующих карт Mora: переписать `titleMeta`, `description`, `result.dayVariants`, тон, структуру `preview/full` или голос Моры.

Эта директива не про генерацию новых изображений и не про добавление новых ID. Для новых карт используй `04-add-tarot-cards.md`.

## Предусловия

Перед началом прочитай:

1. `PROJECT_CONTEXT.md`
2. `docs/project.md`
3. `docs/architecture.md`
4. `docs/design-system.md`
5. `docs/card-style.md`
6. `docs/decisions.md`

Затем открой:

1. `public/assets/cards.js`
2. `src/lib/tarot.ts`
3. `/qa/cards` в активном окружении, если нужно сверить порядок и текущие тексты.

## Главный принцип

Работай короткими пачками и не интегрируй тексты без ручного аппрува автора.

1. Один проход = 3 карты максимум, если пользователь явно не просит меньше.
2. Иди в порядке QA: от старых/первых карт к новым, если пользователь не задал другой порядок.
3. Сначала подготовь текстовые варианты для чтения автором.
4. Не меняй `public/assets/cards.js`, пока автор не принял конкретные варианты.
5. Не меняй `id`, `image`, `num`, `symbol`, `visualHint` и ассеты.
6. Не переписывай рабочую механику вытягивания, snapshot, journal или recent cards.

## Формат каждого обновления

Для каждой карты подготовь:

- `id`
- русское название
- текущую роль карты в daily-card retention
- при необходимости новый `description`
- при необходимости новый `result.titleMeta`
- 3-4 варианта `result.dayVariants` в формате `{ preview, full }`

Требования к `dayVariants`:

- `meaningLabel` для нового формата: `Что это значит сегодня`;
- `preview` — 1-2 абзаца, самостоятельное чтение карты дня;
- `full` — точное начало `preview` плюс 1-3 дополнительных абзаца ниже;
- `preview` должен быть точным префиксом `full`;
- классический смысл Rider-Waite-Smith вплетён в reading, а не вынесен в отдельный справочный блок;
- в тексте считывается персонаж Моры: можно использовать `Мора говорит`, `Мора советует`, `Мора считает`, если это звучит естественно;
- имя Моры не повторяется механически в каждом абзаце;
- варианты различаются дневным сценарием, а не только словами;
- без эзотерического шума, маркетингового тона, эмодзи и обещаний.

## Аппрув

Покажи автору тексты до интеграции.

Для каждого варианта автор может:

- принять;
- попросить правку;
- удалить вариант;
- попросить другой сценарий дня;
- отметить повтор с другой картой.

Карта считается готовой к интеграции только когда автор явно принял все варианты, которые должны попасть в `cards.js`.

## Интеграция после аппрува

После аппрува:

1. Обнови `public/assets/cards.js`.
2. Не добавляй отдельные таблицы текстов в React-компоненты.
3. Проверь, что `src/lib/tarot.ts` адаптирует новый формат без дублей.
4. Проверь `/qa/cards`: варианты видны, `full` отличается от `preview` там, где нужно.
5. Проверь result-screen для одной карты из пачки через debug-параметры.
6. Запусти `npm run lint`.
7. Запусти `./node_modules/.bin/tsc --noEmit`.
8. Если менялась логика или много карт — запусти `npm run build`.

## Фиксация контекста

После принятой пачки обнови:

1. `docs/decisions.md` — какие карты обновлены, почему такой подход, кто принял.
2. `docs/features.md` — только если меняется количество готовых/принятых карт или статус колоды.
3. `docs/card-style.md` — только если появился новый текстовый принцип.

## Граница

- Не коммить и не деплой без отдельного аппрува пользователя.
- Не меняй изображения и ID карт в этой директиве.
- Не объединяй актуализацию текстов с редизайном экрана, багфиксом или добавлением новых карт.
