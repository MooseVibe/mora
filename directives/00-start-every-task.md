# Директива 00 · Start Every Task

> **Правило времени:** если задача по предварительной оценке займёт больше 10 минут, до начала работы предупреди автора и предложи разбить её на шаги. Не оставляй автора без статус-апдейта дольше 2 минут; если команда или процесс зависли либо превысили ожидаемое время, останови их сам и сразу сообщи, что произошло.

## Когда использовать

Открывай эту директиву перед любой задачей в Mora: баг, редизайн, новая фича, новая карта, ревью, документация или деплой.

Это входной фильтр перед остальными директивами. После него выбери одну конкретную директиву из `directives/` и не смешивай несколько задач в один проход.

## Цель

Делать меньше лишнего кода и меньше лишних действий, не снижая качество Mora.

Для этого каждая задача проходит через Ponytail: сначала ищем самый простой рабочий путь, потом меняем только то, что нужно.

## Обязательный порядок

1. Прочитай `PROJECT_CONTEXT.md`.
2. Если задача затрагивает продукт, код или визуал, прочитай:
   - `docs/project.md`
   - `docs/architecture.md`
   - `docs/design-system.md`
3. Если задача затрагивает карты, тексты или генерацию арта, прочитай:
   - `docs/card-style.md`
4. Если задача затрагивает тексты, голос Mora, `description`, `preview/full/share`, `streetText` или `streetVariants`, прочитай:
   - `docs/banned-phrases.md`
5. Прочитай эту директиву.
6. Выбери одну рабочую директиву:
   - `01-redesign-screen.md`
   - `02-add-feature.md`
   - `03-fix-bug.md`
   - `04-add-tarot-cards.md`
   - `05-update-card-texts.md`
   - `06-commit-and-deploy.md`
   - `07-process-feedback.md`
7. Перед изменениями проверь `git status --short`.
8. Перед изменениями сформулируй коротко:
   - что именно делаем;
   - какой самый маленький безопасный шаг;
   - что пользователь сможет проверить глазами.

## Ponytail-правила для Mora

Используй Ponytail в режиме `full` по умолчанию.

Перед тем как писать код или генерировать большой артефакт, пройди лестницу:

1. Это точно нужно сейчас, или можно не делать?
2. Это уже есть в проекте?
3. Это умеет браузер, CSS, Next.js, Supabase или стандартная библиотека?
4. Это решает уже установленная зависимость?
5. Можно ли сделать одной маленькой правкой?
6. Только потом пиши минимальный новый код.

## Что Ponytail не отменяет

Ponytail не разрешает срезать углы там, где Mora требует качества.

Нельзя упрощать за счёт:

- авторизации и сохранения карты;
- правила «одна карта в день»;
- pending draw после логина;
- accessibility и focus-состояний;
- mobile/desktop проверки;
- ритуальных анимаций;
- дизайн-системы;
- канона Rider-Waite-Smith для классических карт;
- ручного аппрува визуала, `preview/full` и `share`;
- `docs/decisions.md` после значимых решений.

## Как работать маленькими шагами

Для каждой задачи:

1. Один проход = один понятный результат.
2. Не делай «заодно».
3. После каждого шага скажи пользователю:
   - что изменилось;
   - где открыть локально;
   - что проверить глазами;
   - продолжать ли следующий шаг, если есть риск.
4. Если шаг начинает разрастаться, остановись и предложи более короткий путь.

## Для добавления карт

Даже в Ponytail-режиме одна новая карта остаётся полным циклом качества:

1. выбрать одну карту;
2. сверить канон;
3. сгенерировать один визуал;
4. получить визуальный аппрув;
5. только потом интегрировать ассет и код;
6. отдельно получить аппрув текстов `preview/full/share`;
7. проверить `/qa/cards`;
8. запустить `npm run lint` и при финальной интеграции `npm run build`.

Упрощать можно процесс вокруг карты, но не саму приёмку карты.

## Хорошая финальная форма ответа

Коротко:

- что сделано;
- какие файлы изменены;
- как проверить;
- какие проверки прошли;
- что осталось, если задача не завершена.

Без длинной защиты решения. Если объяснение длиннее изменения, возможно, изменение слишком сложное.
