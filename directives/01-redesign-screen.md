# Директива 01 · Redesign Screen

## Когда использовать

Пользователь говорит «давай отредизайним экран X» или «сделай UI для экрана X». Механика уже работает, нужно поменять только визуал.

---

## Предусловия

Перед тем как лезть в код — прочитай:
1. `docs/project.md` — чтобы понимать продуктовый контекст
2. `docs/design-system.md` — чтобы держать визуальный язык
3. `docs/features.md` — чтобы знать что на этом экране должно работать

Если какого-то файла нет или он пустой — скажи пользователю и не начинай работу.

---

## Шаги

### Шаг 1 · Узнай контекст экрана

Спроси пользователя (если не сказано):
1. **Какой именно экран?** Точное название и путь (например, `/dashboard`).
2. **Что не нравится в текущем виде?** Что нужно поменять — компоновку, стили, иерархию, добавить новые блоки?
3. **Есть ли референсы или мудборд?** Скриншоты, ссылки на похожие интерфейсы.

Один блок вопросов. Не вываливай 10 штук.

### Шаг 2 · Открой текущий код экрана

Зайди в файл экрана и **прочитай его целиком**. Запомни:
- Какие компоненты используются
- Какая логика (хуки, стейт, API-вызовы)
- Какие данные приходят и как они отображаются

**Не переписывай логику.** Меняешь только разметку и стили. Если логику всё-таки нужно тронуть — спроси разрешения.

### Шаг 3 · Опиши план редизайна

Прежде чем писать код — напиши пользователю короткий план:
1. Какая будет структура экрана (блок 1, блок 2, блок 3)
2. Какая иерархия (что главное, что второстепенное)
3. Какие компоненты из дизайн-системы используешь, какие надо создать новые
4. Что меняется на mobile vs desktop

Спроси: «Идём?». Жди подтверждения.

### Шаг 4 · Делай поэтапно

Не пиши все 300 строк за один раз. Иди блоками:
1. **Структура** — сначала разметка без стилей, чтобы пользователь увидел иерархию
2. **Стили** — потом стили из дизайн-системы
3. **Состояния** — loading, empty, error
4. **Адаптив** — проверка на mobile и desktop

После каждого блока — короткий чек-ин «готов следующий, продолжаю?».

### Шаг 5 · Проверь обязательное

Перед тем как сказать «готово»:

- [ ] Все цвета — через токены из `design-system.md`, никаких хексов в JSX
- [ ] Все отступы — по шкале (4/8/16/24/32/48/64), никаких странных чисел
- [ ] Типографика — серифом только заголовки
- [ ] Есть состояния: loading, empty, hover (если применимo), focus
- [ ] **Анимации продуманы:** переходы между экранами, появление контента, ховеры. Если на экране есть карта — её появление и переворот должны быть ритуальными, не cut. Loader — тематический, не спиннер. Тайминги из `design-system.md`.
- [ ] Работает на 375px и 1440px без поломок
- [ ] Тон голоса в текстах — мистический, не маркетинговый
- [ ] Все рабочие механики экрана продолжают работать (вытягивание, дневник и т.д.)
- [ ] Уважён `prefers-reduced-motion` — для пользователей с этой настройкой анимации упрощены, но не убраны полностью

### Шаг 6 · Зафиксируй

1. Если ввёл новый токен, паттерн или компонент — допиши в `docs/design-system.md`.
2. Допиши строчку в `docs/decisions.md`: «Редизайн экрана X — что поменяли и почему».

---

## Граница

Если пользователь просит:
- **Поменять механику** (например, разрешить вытягивать карту 2 раза в день) → это **не редизайн**, это новая фича. Открой `02-add-feature.md`.
- **Починить баг** → открой `03-fix-bug.md`.
- **Запустить аналитику, А/Б-тесты** → этого в проекте сейчас нет, скажи прямо.

Не объединяй задачи в один поход.
